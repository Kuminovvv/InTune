# Вклад в проект

## Правила качества

### 1. Принципы
- **Простота и явность**: минимальная вложенность, читаемые имена, отсутствие «магии».
- **Одна ответственность**: модуль/файл/функция делают что-то одно.
- **Явные контракты**: чёткие входы/выходы, проверка граничных условий.
- **Воспроизводимость**: детерминированные функции, отсутствие скрытых эффектов.
- **Измеряемая оптимизация**: сначала профилирование/метрики, потом оптимизация.

### 2. Структура проекта
- Чёткая модульность по домену (например: `core/`, `domain/`, `infra/`, `ui/`, `tests/`).
- Внешние границы (API/БД/IO) отделены от бизнес-логики.
- Публичное API каждого модуля минимально: экспортируй только необходимое.

### 3. Именование и стиль
- Понятные, предметные имена: `minDailyRate`, `calculateOverpayment`, `fetchUser`.
- Константы и единицы измерения явны: `ratePctPerDay`, `amountRub`.
- Единый стиль кода обеспечивается линтером/форматтером.

### 4. Обработка ошибок и логирование
- Ошибки — данные: возвращай/бросай информативные ошибки, не «глуши».
- Логируй только то, что помогает отладке (уровни: error/warn/info/debug).
- Не пиши в логи секреты (ключи/токены/персональные данные).

### 5. Производительность и память
- Учитывай асимптотику: избегай квадратичных мест на больших массивах/таблицах.
- Не дублируй крупные структуры, не создавай объекты в горячих циклах.
- Кэшируй дорогое (но с инвалидацией), используй стриминг/пагинацию.

### 6. Тестирование
- Юнит-тесты на чистые функции и критичные ветвления.
- Интеграционные — на границах: БД, API, очереди, файловая система.
- «Баг → тест → фикс». Покрывай регрессии.
- Фикстуры маленькие, детерминированные; мокай внешние зависимости.

### 7. Безопасность и конфиденциальность
- Конфиги и секреты только через переменные окружения, `.env.example` вместо `.env`.
- Валидация входных данных (тип/диапазон/формат).
- Защита от инъекций (SQL/команд/HTML), экранирование и параметризация запросов.

### 8. Документация
- README на русском: запуск, конфиги, архитектура, примеры.
- Комментарии отвечают на «почему», а не на «что»; сложные алгоритмы сопровождай ссылками/доказательствами.
- ADR (по желанию) для значимых архитектурных решений.

### 9. Git-практики
- Ветки: осмысленные (`feature/...`, `fix/...`, `refactor/...`).
- Коммиты мелкие и предметные. Формат:
  ```
  <тип>: кратко и по делу [#issue]

  детали при необходимости
  ```
- PR: «что сделано», «как проверить», чеклист (линт/тесты/миграции/доки).

### 10. Чеклист автора перед пушем
- [ ] Имена и структура понятны без контекста.
- [ ] Функции короткие, цикломатическая сложность разумная.
- [ ] Нет «временного» кода (`console.log`, `print`, закомментированный мусор).
- [ ] Форматтер/линтер/типы/тесты — зелёные.
- [ ] Секреты не попадают в VCS; обновлён `.env.example`.
- [ ] Документация и миграции приложены (если нужно).

## Языковые дополнения

### JavaScript/TypeScript
- `tsconfig`: `"strict": true`, избегай `any` (кроме обёрток), явные типы в публичных API.
- Разделяй UI/бизнес/infra: хук/сервис/адаптеры, не смешивай в компонентах.
- ESLint + Prettier + import-order; соблюдай правила hooks/exhaustive-deps.
- Асинхронность: таймауты, отмена (AbortController), повтор с бэкоффом.
- Большие списки — виртуализируй; ключи в списках — стабильные id.

### Python
- Стиль: PEP8 (flake8/ruff) + black, типизация `mypy` (строгие режимы).
- Разделяй домен/инфраструктуру; для IO — контекстные менеджеры.
- Обрабатывай ошибки собственными исключениями, не перехватывай «всё».
- Производительность: генераторы, итераторы, `dataclasses`/`pydantic` для схем.
- Веб/скрипты: зависимости фиксируй в `requirements.txt`/`poetry.lock`.

### Go
- Ошибки всегда проверяй, оборачивай `fmt.Errorf("context: %w", err)`.
- Интерфейсы минимальны и у потребителя; структуры — с `json`/`db` тэгами.
- Конкурентность: `context.Context`, `errgroup`, не допускай утечки горутин.
- Линтеры: `golangci-lint`; форматирование: `gofmt`/`goimports`.
- БД: `sqlc`/`gorm` с контекстом, транзакции с явными границами.

### SQL
- Явная схема: типы, ограничения, индексы под реальные запросы.
- Запросы параметризованные, без конкатенации строк.
- Миграции атомарные, с откатами; сиды — минимальные и детерминированные.
- Профилируй планы (`EXPLAIN`), следи за N+1 в приложении.

### Bash/CLI
- Скрипты начинаются с `set -Eeuo pipefail`, проверяй зависимости (`command -v`).
- Не храни секреты в истории; передавай через окружение/файлы с ограниченными правами.
- Логи и коды возврата осмысленные; используй `trap` для очистки временных файлов.

### HTML/CSS
- Семантическая разметка, доступность (aria-атрибуты, контраст, фокус).
- Модульность стилей (BEM/CSS-модули/Tailwind), без «магических» чисел.
- Изображения оптимизированы, адаптивная типографика, поддержка `prefers-reduced-motion`.

## Поток работы с git
- Ветки: feature/…, fix/…, refactor/… — осмысленные имена.
- Коммиты и PR — на русском.
- Перед PR: линтер/форматтер/типы/тесты — ок.

## Секреты и конфиги
- `.env` не коммитим; используем `.env.example`.
- Переменные окружения документируем в README.

## Тесты
- Юнит — на чистые функции.
- Интеграционные — на границы (БД/API/файлы).
- Баг → тест → фикс.
